# تیم ایجنتیک AI-Amooz — دفترچهٔ تیم

این پوشه («پوشهٔ ایجنت‌ها») **تیم دائمی ۱۶ نفرهٔ توسعهٔ محصول** است. مسیر `.claude/agents/` مسیر
استاندارد Claude Code برای ایجنت‌های پروژه است — چون داخل ریپو کامیت شده، این تیم **برای همیشه** همراه
پروژه می‌ماند (روی هر ماشین و هر سشن). هر فایل `*.md` یک عضو تیم است: نقش، دانش پروژه، قواعد سخت‌گیرانه
و پروتکل همکاری‌اش.

> اصل بنیادین تیم: **افزایش کیفیت کل اپ + ساخت فیچرهای سخت، با مستندسازیِ همیشه دقیق و حلقهٔ مشورتی.**

## چطور از تیم استفاده کنیم

**۱) درخواست مستقیم** — در گفتگو صریح بگویید؛ مثال‌ها:
- «این تغییر رو بده `code-reviewer` بازبینی کنه»
- «با `security-auditor` این endpoint جدید رو ممیزی کن»
- «`debugger` این خطا رو ریشه‌یابی کنه»

**۲) `/council <موضوع>`** — حلقهٔ مشورتی: ۳ تا ۵ متخصصِ مرتبط موازی نظر می‌دهند، `tech-lead` جمع‌بندی
و داوری می‌کند، و اگر تصمیم معماری باشد ADR ثبت می‌شود. برای تصمیم‌های مهم قبل از کدنویسی.

**۳) `/feature-cycle <شرح فیچر>`** — چرخهٔ کامل ساخت فیچر با گیت‌های اجباری:
اسپک (PM) → طراحی (tech-lead + متخصص‌ها) → ساخت (مهندس‌ها) → تست (QA) → بازبینی (reviewer + security)
→ مستند (writer) → انتشار (release-manager).

### قانون هزینهٔ توکن (مهم)
- ایجنت‌ها **فقط با درخواست صریح کاربر** (یا از طریق دو دستور بالا) اجرا می‌شوند — هرگز فن‌اوت خودسرانه.
- پیش‌فرضِ کارهای عادی: سشن اصلی خودش کار می‌کند و فایل‌های همین پوشه را به‌عنوان **چک‌لیست تخصصی
  رایگان** می‌خواند (خواندن فایل ≈ صفر هزینه؛ اجرای ایجنت = هزینهٔ واقعی).
- `/council` حداکثر ۵ ایجنت؛ `/feature-cycle` بین مراحل گزارش می‌دهد و حداکثر ۲ دور اصلاح بدون تأیید کاربر.

## حلقهٔ مشورتی (پروتکل همکاری)

### هندآف استاندارد — پایان گزارشِ هر ایجنت، بدون استثنا:
```
Decisions: تصمیم‌ها + چرایی (هر کدام یک خط)
Files:     مسیرهای دقیقِ تغییریافته/بررسی‌شده
Docs:      داکِ به‌روزشده — یا بدهی مستندی که ایجاد شد و چرا (سکوت ممنوع)
Risks:     چه چیزی ممکن است بشکند؛ چه چیزی راستی‌آزمایی نشد
Consult next: کدام هم‌تیمی + دقیقاً چه سوالی از او
```

### ماتریس مشورت اجباری
| تغییر | مشاور اجباری |
|-------|--------------|
| auth / پرمیشن / تننسی سازمانی | `security-auditor` (+ تست منفی توسط `qa-engineer`) |
| اسکیما / مایگریشن | `database-engineer` + امضای `tech-lead`؛ دیپلویش → `release-manager` |
| هر چیز LLM (پرامپت/مدل/قرارداد JSON) | `ai-engineer` + اجرای `test_prompts_contract.py` |
| API که فرانت مصرف می‌کند | توافق قرارداد بین `backend-engineer` و `frontend-engineer` قبل از کد |
| UI / کپی فارسی | `ux-designer` (میکروکپی طبیعی — درس «کد سازمان آموزشیِ مدرسه») |
| کارایی / کش | `performance-engineer` (با عدد قبل/بعد) |
| هر دیف غیرجزئی پیش از کامیت | `code-reviewer` |
| هر پوش | گیتِ `release-manager` |

### اصل اختلاف علنی
هر ایجنت موظف است اگر دستور/طرح با CLAUDE.md، گراف معماری یا واقعیت کد در تضاد بود، **علناً مخالفت کند**
— اجرای خاموشِ دستورِ غلط، نقض پروتکل تیم است. داور نهایی: `tech-lead`؛ تصمیم‌های محصولی/برگشت‌ناپذیر: کاربر.

## سیاست مستندسازی (قول تیم: همیشه دقیق)

- خانهٔ مستندات: **`docs/`** — `adr/` (تصمیم‌های معماری، شماره‌دار، تغییرناپذیر) · `features/` (یک اسپک
  زنده برای هر فیچر) · `releases/` (یادداشت هر انتشار) · `runbooks/` (درس‌های عملیاتی). سیاست کامل: `docs/README.md`.
- **قانون طلایی: کد و داکش با هم لند می‌شوند** — کاری «تمام» است که مستندش هم در همان سری کامیت باشد.
  خط `Docs:` در هندآف، مکانیزم اجرایی همین قانون است.
- هر واقعیت فقط یک خانه دارد؛ بقیه به آن لینک می‌دهند. هر داک تاریخ و Status دارد. `technical-writer`
  مالک و ممیز این درخت است.

## نگهداری تیم
- `model: inherit` برای همه — ایجنت‌ها همیشه با همان مدلِ سشن اجرا می‌شوند.
- دو ممیز (`security-auditor`, `code-reviewer`) عمداً ابزار نوشتن ندارند — گزارش می‌دهند، دست نمی‌برند.
- عضو جدید/ویرایش نقش: فایل `*.md` با همین قالب (frontmatter: `name`, `description`, `tools`, `model`)
  + به‌روزکردن جدول همین README + ثبت در `docs/adr/` اگر ساختار تیم عوض شد. مالک این فایل:
  `tech-lead` + `technical-writer`.
- درس جدید هر ایجنت (گوچای تازه، شکست تازه) باید به فایل همان ایجنت اضافه شود — تیم نباید یک اشتباه را دو بار بکند.
